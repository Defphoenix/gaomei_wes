# 安装与升级

## 1. 系统要求

- 推荐 Linux x86_64 服务器。
- 推荐 Mamba 2.x；Conda 可用，但依赖求解通常较慢。
- 生产安装需要足够磁盘空间；HLA*LA graph 解压、prepare 和 BWA 索引会占用大量空间。
- HLA*LA `prepareGraph` 建议至少约 40 GB 可用内存。
- macOS 可用于代码开发和小规模模拟测试，不保证 HLA*LA 等 Linux 专用模块完整运行。

## 2. 获取代码

SSH：

```bash
git clone git@github.com:Defphoenix/gaomei_wes.git
cd gaomei_wes
```

HTTPS：

```bash
git clone https://github.com/Defphoenix/gaomei_wes.git
cd gaomei_wes
```

从 Mac 复制到服务器时，兼容系统自带 rsync 2.6.9 的命令为：

```bash
rsync -avP \
  --exclude '.git/' \
  --exclude 'results/' \
  --exclude 'logs/' \
  --exclude '.runtime/' \
  /local/path/gaomei_wes/ \
  user@server:/PUBLIC/project/wes_pipeline/
```

不要使用旧版 macOS rsync 不支持的 `--info=progress2`。

## 3. 使用 Mamba 创建生产环境

```bash
ENV_ROOT=/PUBLIC/gomics/guofenghua/envs/wes

bash scripts/create_conda_envs.sh \
  --env-root "${ENV_ROOT}" \
  --mamba-bin mamba \
  --production
```

生产安装创建以下绝对路径 prefix：

| Prefix | 主要用途 | 关键约束 |
|---|---|---|
| `big_wes_pipeline_env` | QC、BWA、Samtools、GATK、MSI、报告 | Java 17 |
| `wes_vep_env` | Ensembl VEP 115 | Perl/VEP cache 115 |
| `wes_snpeff_env` | SnpEff/SnpSift | Java 21，与GATK隔离 |
| `wes_hla_env` | MHCflurry binding | 模型另行下载 |
| `wes_hla_typing_env` | HLA*LA | Linux；graph另行准备 |
| `wes_cnv_env` | CNVkit、DNAcopy | matched/pooled normal CNV |

Manta 1.6.0 已归档，不属于 `--production` 默认环境。确需兼容 SV 模块时在 Linux
上额外使用 `--with-sv`。

## 4. 加载和验证

```bash
source "${ENV_ROOT}/env.sh"

gatk --version
java -version
bwa 2>&1 | head
samtools --version
bcftools --version
picard -h 2>&1 | grep -m1 PicardCommandLine
bash scripts/run_snpeff_env.sh -version
mamba run -p "${ENV_ROOT}/wes_cnv_env" cnvkit.py version
mamba run -p "${ENV_ROOT}/wes_cnv_env" \
  Rscript -e 'stopifnot(requireNamespace("DNAcopy", quietly=TRUE))'
bash scripts/run_code_tests.sh
```

环境使用绝对 prefix，因此激活也必须使用完整路径：

```bash
mamba activate /PUBLIC/gomics/guofenghua/envs/wes/big_wes_pipeline_env
```

不能只写 `mamba activate big_wes_pipeline_env`。

安装器会将实际解析的软件版本和 build 写入 `${ENV_ROOT}/manifests/`。YML 表示需求，
manifest 才是该服务器本次安装的实际结果。

## 5. MHCflurry 模型

```bash
mamba run -p "${ENV_ROOT}/wes_hla_env" \
  mhcflurry-downloads fetch models_class1_presentation
```

检查：

```bash
mamba run -p "${ENV_ROOT}/wes_hla_env" mhcflurry-downloads info
```

模型需要由运行 MHCflurry 的同一环境管理。Linux 和 macOS 环境通常不能直接共享
整个 Conda prefix，但模型数据可按版本校验后复制。

## 6. HLA*LA graph

准备下载好的 `PRG_MHC_GRCh38_withIMGT.tar.gz`：

```bash
bash scripts/prepare_hlala_graph.sh \
  --archive /path/to/PRG_MHC_GRCh38_withIMGT.tar.gz \
  --reference-dir /path/to/reference_data \
  --env-prefix "${ENV_ROOT}/wes_hla_typing_env"
```

完成标志包括：

```text
serializedGRAPH
extendedReferenceGenome/extendedReferenceGenome.fa.bwt
extendedReferenceGenome/extendedReferenceGenome.fa.sa
```

这些是共享参考资源，只需对同一 graph 准备一次，不应在每个项目中重建。

## 7. 更新

仅更新代码：

```bash
git pull --ff-only origin main
bash -n run_pipeline.sh config.sh scripts/*.sh
bash scripts/run_code_tests.sh
```

同步已有环境到新版 YML：

```bash
bash scripts/create_conda_envs.sh \
  --env-root "${ENV_ROOT}" \
  --mamba-bin mamba \
  --production \
  --update-existing
```

如果新版本增加 config 变量，建议使用最新版项目生成器重新生成项目。FASTQ 可继续
使用软链接，不需要重复复制。

## 8. 常见 Java 问题

- GATK class file 61：应使用 Java 17 主环境。
- SnpEff class file 65：应通过独立 `wes_snpeff_env` 使用 Java 21。
- 不要为了 SnpEff 升级 GATK 主环境 Java。
